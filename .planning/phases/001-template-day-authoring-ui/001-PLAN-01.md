<objective>
Expand `template_day_slots` to support complete operational slot details (route/boat type/capacity + ticket setup constraints + internal notes), and ensure PowerSync downlink/uplink + tests support the new fields end-to-end.
</objective>

<must_haves>
- Slot has an explicit, persisted "ticket setup constraints" payload that the UI can edit offline and upload safely.
- Slot has internal staff notes persisted and synced.
- Existing API/upload flows remain idempotent for updates (re-sending same payload produces same row state).
</must_haves>

<threat_model>
- Assets: staff-only scheduling templates, operational notes.
- Entry points: `POST /api/powersync/upload` (template_day_slots CRUD).
- Threats:
  - Unauthorized write into another program's slot data (mitigate with existing program membership checks; add tests if needed).
  - Malformed JSON payload causing server errors or data loss (mitigate with validation + safe casting + tests).
  - Oversized notes payload impacting storage/perf (mitigate with max length validation).
</threat_model>

      - `ticket_setup` (string|object|null)

      Contract:
      - Client/local schema stores `ticket_setup` as a JSON string (PowerSync `column.text`).
      - Server accepts either:

        - an already-parsed object (defensive; supports future clients).
      - After decode, validate it is an object matching the canonical v1 shape (see `001-RESEARCH.md`), then persist into the DB JSON column.
      - Defaults mode stores `ticket_setup=null` (enforced by validation + tests).

      Ensure validation rejects invalid JSON structures (non-object values for `ticket_setup`) and overlong notes.
      Keep existing authorization behavior (program membership checks) unchanged.
    </action>

      - When posting a PUT payload with these fields, the database row contains matching values
